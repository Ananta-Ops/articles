IDOR, SSRF, and a Very Real Secrets Problem: OWASP in Practice, Not in Theory

securityowaspincidents

It’s easy to treat OWASP Top 10 categories as academic — IDOR, SSRF, broken access control, they sound like exam terms. They aren’t. Every one of them has a recent, named, dated incident behind it, and the fixes are the same few lines of code they’ve always been.

IDOR: the bug that’s just “change a number in the URL”

In June 2025, security researchers Ian Carroll and Sam Curry found they could pull millions of job applicants’ chat transcripts and session tokens out of McDonald’s “McHire” recruitment chatbot — built on the Paradox.ai platform — by simply decrementing an integer in a lead_id parameter. No authentication bypass, no clever exploit chain. Just an object reference that wasn’t checked against the requesting user’s own session. It was fixed the same day it was reported, which tells you how straightforward the fix was — the hard part was that it existed at all, in production, at that scale.

The same bug class, same year: smart car alarm APIs let attackers take over roughly 3 million vehicle accounts, again through unchecked object references. The fix, every time, is the same one-line discipline: every lookup by ID has to verify the requester actually owns that ID, not just that the ID exists.

SSRF: still the fastest way from a web form to your cloud credentials

Server-side request forgery attacks against cloud metadata endpoints reportedly surged over 450% between 2023 and 2024, and 2025 kept the trend going: a Pandoc SSRF vulnerability (CVE-2025-51591) reached instance metadata through a crafted iframe, a ClickHouse SSRF bug abused its url table function, and F5 Labs documented an active March 2025 campaign systematically probing EC2-hosted sites specifically to steal AWS credentials through the metadata service.

The uncomfortable fact underneath all of this: as of recent surveys, only about half of EC2 instances actually enforce IMDSv2 — the metadata-service version that requires a token and can’t be reached through a naive SSRF payload. That’s not a hard fix. It’s a checkbox in an AWS console that a meaningful fraction of production workloads still haven’t checked.

Supply chain: the dependency you didn’t write is still your bug

The npm ecosystem had a rough second half of 2025. The “Shai-Hulud” worm compromised over 180 packages, harvesting GitHub personal access tokens and cloud credentials directly out of CI pipelines and self-propagating to new packages — CISA issued an alert. Separately, the tj-actions/changed-files GitHub Action (CVE-2025-30066) was compromised via a maintainer’s stolen PAT, with the attacker rewriting version tags used across more than 23,000 repositories to dump CI runner memory into public build logs.

Neither of these required your team to write a single vulnerable line. They required your team to depend on something that did — which is exactly why dependency pinning, CI secret scoping, and treating a compromised CI log as a real incident (not a curiosity) all matter more than most teams treat them.

Secrets sprawl: the boring one that just keeps growing

GitGuardian’s most recent annual report counted over 28 million new hardcoded secrets pushed to public GitHub repositories in a single year — a 34% year-over-year increase — with AI-service secrets specifically up over 80%. That’s not a shrinking problem as tooling improves. It’s a growing one, because more code gets written, faster, by more people (and more AI assistants), than ever before.

The pattern underneath every one of these

None of this is exotic. IDOR is “check ownership, not just existence.” SSRF is “validate and restrict outbound URLs from server-side code, and enforce IMDSv2.” Supply chain is “pin dependencies and scope CI secrets tightly.” Secrets sprawl is “scan pre-commit and in CI, and rotate anything that’s ever touched a public surface.” The bug classes are old. The incidents are recent. The fixes have not changed.
